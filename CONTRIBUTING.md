# Contributing / Contribuir

## English

Thanks for helping improve SpecShowdown.

### Before you start

- Open an issue for large changes so we can align on scope.
- Keep pull requests focused and reviewable.
- Update both English and Spanish copy when UI text changes.

### Local workflow

1. Fork the repository and create a branch.
2. Install dependencies with `npm install`.
3. Run the app with `npm run dev`.
4. Build before opening a PR with `npm run build`.

### Code standards

- Keep the project static-friendly and fast on low-powered devices.
- Prefer accessible, semantic HTML and keyboard-friendly interactions.
- Keep local-storage behavior privacy-safe and explicit.
- Avoid adding backend coupling unless it remains optional and clearly separated.

### Commit style

Public history should stay easy to scan. Preferred prefixes:

- `feat:` for user-facing features
- `fix:` for bug fixes
- `docs:` for documentation changes
- `chore:` for maintenance
- `refactor:` for internal cleanups with no user-facing change

### Pull requests

- Explain the problem and the resulting behavior.
- Include screenshots or short recordings for UI changes.
- Mention any follow-up work that is intentionally out of scope.

## Español

Gracias por ayudar a mejorar SpecShowdown.

### Antes de empezar

- Abre un issue para cambios grandes y así alineamos el alcance.
- Mantén los pull requests enfocados y fáciles de revisar.
- Actualiza el contenido en inglés y español cuando cambie el texto visible de la UI.

### Flujo local

1. Haz un fork del repositorio y crea una rama.
2. Instala dependencias con `npm install`.
3. Ejecuta la app con `npm run dev`.
4. Compila antes de abrir un PR con `npm run build`.

### Estándares de código

- Mantén el proyecto amigable con hosting estático y rápido en equipos modestos.
- Prefiere HTML semántico, accesible y con soporte de teclado.
- Mantén el uso de almacenamiento local claro y respetuoso con la privacidad.
- Evita acoplar un backend salvo que siga siendo opcional y claramente separado.

### Estilo de commits

Para que el historial público sea fácil de leer, se recomiendan estos prefijos:

- `feat:` para funciones visibles al usuario
- `fix:` para correcciones
- `docs:` para cambios de documentación
- `chore:` para mantenimiento
- `refactor:` para limpiezas internas sin cambio visible

### Pull requests

- Explica el problema y el comportamiento final.
- Incluye capturas o grabaciones cortas cuando cambie la UI.
- Menciona cualquier trabajo pendiente que quede fuera de alcance.
